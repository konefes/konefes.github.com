Pedal Portal - In Progress

Portable video and data system for assessing rider locomotion and behavior

Description:

  The goal of this project is to adapt a current GPS-enabled camera system by engineering a new solution that facilitates greater ease of use and reliability. The system will be used to measure the activity and behavior of a bicyclist by providing information on the length, distance, and route infrastructure utilized on the trip. As a team of four, we need to redesign the current system (which is built from three separated, commercial cameras) to provide a more efficient and informative solution.

Details:

  The core of the bike hardware consists of an array of three Raspberry Pis, each capturing footage from a camera module. The feeds are recorded at 1280x720/30 fps, and stored on a usb flash drive connected to the master Pi. A synced GPS unit provides accompanying location data.

  Riders indicate safety-critical events with a push button, which marks the timing of the event on a log that automatically syncs with the captured video when the project is imported to the software.

  Power will be supplied by either one or two USB battery packs, depending on results from testing once a prototype is in a testable state.

  • Tools: C#, C, Raspberry Pi w/ Raspbian OS

Real-time Remote Temperature Sensor

Rails web app for monitoring past and present sensor readings

Description:

  This project called for the development of a compact, battery-powered box capable of reading temperature and transmitting that data to server. A client was also developed to provide a graphical interface for the data as well as remote controls for the sensor box.

Details:

  The temperature sensor box is controlled by an Arduino Uno R3, which runs a continuous loop with a half second timer to check for temperature data and send an HTTP GET request with appended temperature data over the Ethernet connection if available. The DS18B20 temperature sensor operates using the OneWire protocol, so the Arduino OneWire library is used to interpret data and send control signals. Additionally, the DallasTemperature library created by Miles Burton is used in conjunction with the OneWire library to facilitate easier communication syntax.

  Upon power-up, the box attempts to initialize a network connection over the Ethernet port on the Arduino Ethernet shield. If successful, an HTTP GET request is sent to the Rails server each time the timer expires (using the same timer as the temperature sensor readings). The temperature data obtained from the sensor is appended to the GET request, and the server sends back a bare text HTML page that only includes LED toggle control information.

  Once temperature data is received by the Rails server, it is stored in a SqLite database.. As the page loads a loop runs that checks the board’s connection, checks if the sensor is attached (receiving temp data), checks for phone information to be entered, and updates the graph. When updating the graph, ajax calls are run to check for the data in the database. The graph is created using the Highcharts api, which updates the data as it is received. While the graph is updated the program checks to see if the high or low temp limits have been reached and uses a post call to run the ruby function to send a text alert.

  • Date: Fall 2015

  • Design Documentation on Google Docs

  • Source Code on GitHub

  • Tools: Arduino, C++, Ruby on Rails

Signal Decoder

Arduino-based audio packet decoder

Description:

  The objective of this laboratory was to develop a device can decode audio frequency-shift keyed (AFSK) packets encoded at 1200 baud according to the Bell 202 standard with NRZI (Non-Return to Zero, Inverted) encoding

Details:

  As a team of four, we designed the hardware and software necessary to determine the local frequency of the AFSK signal consisting of a train of 1200 Hz and 2200 Hz tones representing bits in a message to be communicated. The hardware consisted of a level shifter and inverter circuit, which supplied a clear NRZI signal to the Arduino decoder.

  We developed an application for the Arduino decoder to translate HDLC packets that have been sent using NRZI signaling with the Bell 202 tones. The application used the Arduino’s serial communication to display the contents of each packet received.

  • Tools: Arduino, C++

Secure Messenger

Encrypted messaging app using the Off-The-Record (OTR) protocol

Description:

  As a team of four, we were tasked with developing an instant messaging client that supports strong end-to-end encryption so that only the end users can decrypt and read the messages exchanged. The end-to-end encryption technique used was required to provide for authentication of users via the socialist millionaire protocol. We selected the OTR protocol and developed a Java application to implement it.

Details:

  OTR is a secure messaging protocol that allows you to have private conversations over IM by providing encryption, authentication, deniability, and perfect forward secrecy. No one else can read your instant messages and you are assured the person you are talking to is who you are wanting to correspond with. End-to-end encryption uses Diffie-Hellman key exchange, and authentication is covered by using the Socialist Millionaire Protocol (SMP).

  This OTR library uses the Socialist Millionaire Protocol (SMP) to prevent man-in-the-middle attacks and eavesdropping without requiring users to work with complex fingerprints. This SMP implementation provides the convenience of not manually comparing public key fingerprints through an outside channel, allowing a relatively weak password/passphrase in natural language that is easier to use and understand. SMP runs by exchanging 2 secret strings, for example “x” and “y”. Then a series of message is exchanged to reveal the value if x=y without revealing any other information regarding the two secret strings or the exact value. This allows the users to determine whether the same secret information is shared with no danger of revealing the secret to an attacker.

Health E-Records

Hospital management software for scheduling, billing, payroll, etc.

Description:

  Developed with PHP using CodeIgniter, this web app provides doctor, patient, nurse, and admin profiles along with associated functionality. The app streamlines a hospital’s records and eliminates the need for physical paperwork.

Details:

  From the patient homepage, links provide easy access to functionality that includes Schedule an Appointment, Show My Appointments, Fill Out Medical Record, Pay Bill, and View Medical Record. These home page links allow a user to fill out their own medical record forms, view the current status of their medical records, search and register for appointments with a Doctor, and view/change/cancel their upcoming appointments. A patient receives an email notification if a Doctor changes the time or cancels a scheduled appointment with the patient. Once a doctor has determined the user’s treatment as “complete,” and an admin has generated a bill, the user can pay their bill by verifying the total amount by typing it in and clicking submit on the bill form view.

  The doctor homepage provides access to Show My Patients, My Info, Salary Info. A doctor is able to view a list of their patients with future appointments, which will link to the medical records/demographic information of the patients. A doctor also has a link to fill out forms for Prescriptions and Diagnosis for the given patient at the time of the appointment. A doctor receives an email if one of their patients changes or cancels an appointment with them, and a doctor may begin the bill generation and payment process by clicking “treatment complete” button for the specific patient. Additionally, a doctor has functionality to display their yearly salary information. This salary is dependent on the number of patients which he sees over the pay period.

  The nurse homepage is a customized version similar to a doctor’s page, and both are able to choose their work schedule and update as needed. Admin profiles have the ability to evaluate all other actor types, approve treatments, and approve and distribute payroll.

Duck Hunt

Classic game clone created with Java

Description:

  A game based on the NES classic Duck Hunt, featuring two modes: duck and clay pigeon.

Details:

  The goal of this project was to implement a classic arcade game in Java with modern twists. With Duck Hunt, the similarity between the two game types allowed utilizing polymorphism to implement the functionality. General actors were created with all the shared functionality, and each specific type of sprite inherited behaviors through multiple levels of the object hierarchy. Ducks were given random behavior, while the clay pigeons followed a familiar arc.

  Each game gave the player three shots per round, gradually increased the difficulty through each round, and tabulated the scores at the end. High scores stored on a server were compared and updated as needed.

Backup Sensor

Sonar distance finder for vehicle assistance

Description:

  This project was my first experience designing an embedded system, and it utilizes an ATMega chip programmed with AVR Assembler using C.

Details:

  This project utilizes a SainSmart HC-SR04 Ranging Detector controlled by an ATMega microcontroller to create a backup sensor for an automobile. An LCD with custom characters is implemented to display the distance and visualization, and a piezo buzzer is also utilized as an auditory alert. The system is designed as a functional implementation for older model vehicles to obtain the sonar-sensing capabilities, so an on-board integrated power converter is included to take 12v DC from the automobile’s reverse light to turn on and supply voltage.

  The HC-SR04 operates by sending out 40kHz pulses and sending an echo back to the ATmega88PA. A timer is utilized to interpret the signal from the range finder, and the information is presented to the user through a visual and auditory alert.

  Version 2 of this project implemented multiple range detectors through the use of shift registers, along with an LED array to visualize both distance and direction of obstacles. This version supplied almost 360 degrees of measurements.

  • Date: Spring 2014

  • Source Code on GitHub

  • Tools: ATTiny, AVR Assembler, C

Calculator Server

Multi-client server with calculator functions

Description:

  The goal of this assignment was to build a calculator service to which multiple clients can connect to perform simple computations.

Details:

  The server is responsible for maintaining a table for each client connections, and each table provides a mapping from variables names and their values. Each new client is stored in a linked list of clients, and each client in that list has its own linked list of variables to keep their information separate from one another. Upon connecting to a new client, the server waits for input. It keeps track of each client's information, and frees up the client memory when a disconnect is detected. The server reads input from the client until it encounters a newline character, and then performs the corresponding actions.

  The Linux operating system provides the select function, which was used for managing connections with multiple clients. It allows the program to monitor multiple sockets for certain activity to occur. For example, if there is some data to be read on one of the sockets, select will provide that information.

  

  • Date: Fall 2015

  • Source Code on GitHub

  • Tools: C, Linux
